her.ai — Talk to Anyone from Your WhatsApp History

python mcp rust ai rag whatsapp cursor

The Idea

What if you could text someone — and an AI replies exactly the way they do? Same slang. Same Hinglish. Same energy. Same emoji game.

Not a generic chatbot that says "As an AI language model...". An AI that actually sounds like that person because it learned from thousands of real WhatsApp messages between you two.

That's her.ai. You export a WhatsApp chat, point the tool at it, and start texting. The AI picks up on the other person's vibe — the way they shorten words, the language switches between Hindi and English, the overuse of "hahaha", all of it.

How It Works

The architecture is surprisingly simple. No fine-tuning, no training, no GPUs. Just smart context engineering.

WhatsApp Export (.txt) ↓ Regex Parser (handles AM/PM, media, deletions) ↓ Style Extractor (last 60 messages from the persona) ↓ History Builder (last 400 messages for context) ↓ MCP Server (builds system prompt, tracks session) ↓ Cursor's LLM (generates reply as the persona)

The core trick: instead of building a RAG pipeline with vector stores and embeddings, I feed the LLM a carefully constructed prompt containing style examples (how the persona actually texts) and recent history (what you've been talking about). The LLM does the rest.

The MCP Server

The interesting part of her.ai is the MCP server — it plugs directly into Cursor IDE as a context provider. No API keys, no rate limits, no external costs. It piggybacks on whatever model Cursor is already running.

What's MCP?

Model Context Protocol — an open standard that lets tools feed context into LLMs. Think of it like a plugin system for AI editors. Your MCP server provides tools that the LLM can call, and the LLM uses the returned context to generate responses.

The server exposes two tools:

chat_as_persona

Send a message, get back the full persona context — style samples, chat history, session memory. Cursor's LLM reads this and replies in character.

list_chats

Lists available WhatsApp exports, shows participants and message counts. Tells you who the persona will mimic.

What It Looks Like

In Cursor, you just type me: followed by your message. A Cursor Rule auto-triggers the MCP tool, the LLM gets the persona context, and replies as that person.

Talking to the persona in Cursor
me: hey kya kar rahi ho

It remembers the conversation across turns. If you mention something from message 1, the reply to message 5 still has that context. Session memory is maintained in-process on the MCP server.

The Rabbit Hole

What looks clean now went through about six different iterations. Here's the journey:

Attempt 1: Direct Gemini API

First try was a Python script hitting Gemini directly with the full chat as context. Problem: the LLM kept analyzing the chat instead of role-playing. It would say things like "Based on the conversation patterns, I observe that..."

Fix: rewrote the system prompt from analytical instructions to direct role-play commands. "You ARE this person. You text like them. Never break character."

Attempt 2: Custom Parser

LangChain's WhatsAppChatLoader couldn't parse the chat format. Indian WhatsApp exports use DD/MM/YY, HH:MM am with lowercase am/pm — the loader expected uppercase and a different date format entirely.

Wrote a custom regex parser that handles the actual export format:

# The regex that actually works with Indian WhatsApp exports pattern = re.compile( r'^[\d/]+,\s+[\d:]+\s+[ap]m\s+-\s+([^:]+):\s*(.+)$', re.IGNORECASE )

Also filters out <media omitted>, deleted messages, and encryption notices — stuff that would confuse the persona.

Attempt 3: Rate Limits Hit

Gemini's free tier has a 20 requests/minute limit. After a few minutes of chatting, you hit 429 ResourceExhausted and have to wait 45 seconds. Not a great conversational experience.

This is when the MCP idea clicked — why pay for API calls when Cursor already has an LLM running? Build a context provider, not a chatbot.

Attempt 4: MCP Server (the one that stuck)

The MCP server doesn't call any LLM itself. It parses the chat, builds a massive system prompt with style examples and history, and returns it as text. Cursor's own model does the generation. Zero API cost. No rate limits.

The key insight

An MCP server is a context provider, not a model. It gives the LLM the right information at the right time. The LLM you already have does the thinking. This flips the economics — instead of paying per token to an external API, you use whatever model your editor already runs.

The Prompt Engineering

The system prompt is where the magic happens. Three sections:

1. Style Samples

The last 60 messages from the persona, presented as examples. The LLM sees exactly how they text — word choice, abbreviations, language mixing, message length. Not a description of their style, but the actual messages.

2. Recent History

The last 400 messages between both people. This gives the LLM relationship context — inside jokes, shared plans, how they address each other, ongoing topics.

3. Session Memory

The current conversation (this session). Each turn gets appended to an in-memory list. When the LLM generates a reply, the Cursor Rule passes it back via previous_persona_reply on the next call, so the server logs it into history.

# Session memory: list of (speaker, message) tuples # Persists across turns within a single MCP server process session_messages: list[tuple[str, str]] = [] # On each call, the previous LLM-generated reply is logged if previous_reply: session_messages.append((responder_name, previous_reply)) session_messages.append((sender_name, message))

Rust + Python

The project is bilingual. Rust handles high-performance chat parsing (converting between WhatsApp export formats), while Python handles the ML/RAG layer and the MCP server.

Why Rust for parsing? WhatsApp exports can be huge — tens of thousands of messages. Rust's regex engine is fast enough to parse them in milliseconds. The Rust binary converts bracket-format exports ([DD/MM/YY, HH:MM:SS]) to the simpler format the Python parser expects.

The Python side uses uv for dependency management — one uv sync and everything's installed. The MCP server runs via uv run python mcp_server.py, which Cursor launches automatically from the config.

Alternative Approaches

her.ai includes three other RAG approaches beyond the MCP server, in case you want to experiment:

Gemini Direct

Full chat history as context, direct prompting. Works well but hits rate limits on free tier.

Ollama + FAISS

Fully local RAG. No API keys, no internet. Uses Ollama for embeddings and generation, FAISS for vector search.

OpenAI + Chroma

Cloud RAG with OpenAI embeddings and Chroma vector store. Better retrieval quality, but costs money.

The MCP approach won because it's the only one that's free, fast, and integrated into the editor. No context switching. No separate terminal. Just chat in your IDE.

Setup

What you need

Cursor IDE — or any editor that supports MCP.
Python 3.10+ — with uv for package management.
A WhatsApp chat export — exported as .txt without media.

# Clone and install git clone https://github.com/rootCircle/her.ai cd her.ai/whatrag uv sync # Export a WhatsApp chat and drop it in cp ~/Downloads/WhatsApp\ Chat.txt ../chat_files/chat.txt # Copy and edit MCP config for Cursor cp .cursor/mcp.sample.json .cursor/mcp.json # Set SENDER_NAME to your name as it appears in the chat

Restart Cursor. The MCP server starts automatically. Type me: hey in Cursor's chat and the persona replies.

The Hard-Won Lessons

  • Prompt > Architecture. I tried RAG, vector stores, and embeddings. In the end, a well-crafted system prompt with raw chat examples outperformed all of them for persona mimicry.
  • Parsers are never simple. WhatsApp exports differ by locale, OS, and version. The regex that works for one person's chat breaks on another's. Case-insensitive matching and flexible date patterns saved the day.
  • MCP flips the cost model. Instead of paying per API call, you build a context provider and let the editor's built-in model do the work. Same quality, zero marginal cost.
  • Session memory matters. Without it, every message is a cold start. The persona forgets what you just said. A simple in-memory list of (speaker, text) tuples fixed it.
  • Style > Knowledge. For persona simulation, matching how someone texts matters more than what they know. 60 raw message examples teach the LLM more about someone's vibe than any description could.

What's Next
